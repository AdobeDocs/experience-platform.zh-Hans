# 数据治理概述

Adobe Experience Platform的核心功能之一是将来自多个企业系统的数据汇集在一起，以便更好地让营销人员识别、了解和吸引客户。 贵组织或法律法规可能会对此数据设置使用限制。因此，请务必确保您的数据操作符合以下条件： [!DNL Platform] 符合数据使用策略。

Adobe Experience Platform数据管理允许您管理客户数据，并确保遵守适用于数据使用的法规、限制和策略。 它在以下方面发挥着关键作用 [!DNL Experience Platform] 在不同的级别，包括编目、数据谱系、数据使用标签、数据使用策略和控制营销活动的数据使用。

>[!NOTE]
>
>在Experience Platform中，数据治理只关心数据的使用或激活方式，而不管用户是否正在执行该操作。 有关如何控制组织中特定Platform用户访问特定数据字段的信息，请参阅以下文档： [基于属性的访问控制](../access-control/abac/overview.md) 而是。

## 数据治理角色

作为一个概念，数据治理既不是自动的，也不是在真空中发生的。 最初只是一个人的角色，通常被认为是数据管理员，但随着数据治理生态系统的扩展，该角色已显着增长。 如今，数据治理需要持续管理和监控才能取得成功，并且依赖于数据管理员拥有一些工具，通过这些工具可以正确标记数据、创建使用策略以及强制遵守这些策略。

虽然数据治理应该是组织中每个人的责任，但以下是数据治理周期中的一些基本角色：

![数据治理角色](./images/overview/roles.png)

### 数据管理员

数据管理员是数据管理的核心。 此角色负责解释法规、合同限制和政策，并将其直接应用于数据。 根据他们对这些法规、限制和政策的理解，数据管理员的角色包括：

* 审查数据、数据集和数据示例，以应用和管理元数据使用标签。
* 创建数据策略并将其应用于数据集和字段。
* 将数据策略传达给组织。

### 营销人员

营销人员是数据治理的终点。 他们从数据管理员、科学家和工程师创建的数据治理基础架构中请求数据。 营销人员可在营销伞状结构下包含许多不同的专业，其中包括：

* 营销分析人员需要数据才能了解个人和群组（也称为区段）客户。
* 营销专家和体验设计人员使用数据设计新的客户体验。


## 数据治理框架

数据治理框架简化并简化了数据分类和创建数据使用策略的流程。 应用数据标签并设置数据使用策略后，可以评估营销操作以确保正确使用数据。

数据管理框架有三个关键元素：标签、策略和实施。

1. **标签：** 对反映与隐私相关的注意事项和合同条件的数据进行分类，以符合法规和组织策略。
1. **策略：** 描述允许或不允许对特定数据执行哪种营销操作。
1. **强制执行：** 使用策略框架在不同数据访问模式间提供建议并强制实施策略。

## 数据使用情况标签

数据管理允许数据管理员在架构字段级别和数据集级别应用使用标签，以便根据应用的策略类型对数据进行分类。

数据治理框架包括预定义的数据使用标签，可用于通过三种方式对数据进行分类：

![数据使用情况标签类别](./images/overview/label-categories.png)

* **合同“C”数据标签：** 标记和分类具有合同义务或与客户数据治理策略相关的数据。
* **“I”身份数据标签：** 为可识别或联系特定人员的数据加上标签并进行分类。
* **敏感的“S”数据标签：** 标记和分类与敏感数据（如地理数据）相关的数据。

>[!NOTE]
>
>请参阅指南，网址为 [支持的数据使用标签](labels/reference.md) 以获取可用标签的完整列表，以及每种标签类型的定义。

可以随时应用标签，从而灵活地选择管理数据的方式。 最佳实践鼓励在将数据摄取到其中后立即为其设置标签 [!DNL Experience Platform]，或当数据在中可用时 [!DNL Platform].

请参阅概述，位于 [数据使用标签](./labels/overview.md) 了解更多信息。

## 数据使用策略 {#data-usage-policies}

为了使数据使用标签有效地支持数据合规性，必须实施数据使用策略。 数据使用策略是描述允许或限制您对中的数据执行的营销操作类型的规则 [!DNL Experience Platform].

营销操作的示例可能是希望将数据集导出到第三方服务。 如果制定了声明个人身份信息(PII)无法导出的策略，并已将“I”标签（身份数据）应用于字段级别或数据集， [!DNL Policy Service] 阻止任何会将此数据集导出到第三方目标的操作。 如果发生其中一次操作尝试，策略服务将发送一条消息，告知您违反了数据使用策略。

有两种可用的策略：

* **[!UICONTROL 数据治理策略]**：根据所执行的营销操作和相关数据携带的数据使用标签，限制数据激活。
* **[!UICONTROL 同意政策]**：筛选可以激活的用户档案 [目标](../destinations/home.md) 基于客户的同意或偏好。

应用数据使用标签后，数据管理员可以使用 [!DNL Policy Service] API或 [!DNL Experience Platform] 用户界面。 有关数据使用策略和营销操作的更多信息，请参阅 [策略概述](./policies/overview.md).

>[!IMPORTANT]
>
>默认情况下，所有数据使用策略(包括Adobe提供的核心策略)都处于禁用状态。 为了考虑实施单个策略，您必须手动启用该策略。

### 数据治理术语 {#data-governance-terminology}

下表概述了与数据治理和数据治理框架相关的关键术语。

| 搜索词 | 定义 |
|---|---|
| **合同标签** | “C”合同标签用于对具有合同义务或与组织的数据治理策略相关的数据进行分类。 |
| **跨站点数据** | 跨站点数据是来自多个站点的数据的组合，包括站点内数据和站点外数据的组合或来自多个站点外来源的数据的组合。 |
| **数据治理** | 数据治理包括所使用的战略和技术，以确保数据在使用方面遵守法规和公司政策。 |
| **数据管理员** | 数据管理员是负责管理、监督和强制执行组织数据资产的人员。 数据管理员还可以确保数据治理策略受到保护和维护，以遵守政府法规和组织政策。 |
| **数据使用情况标签** | 通过使用数据使用标签，用户能够对反映与隐私相关的考虑因素和合同条件的数据进行分类，以符合法规和公司政策。 |
| **数据集标签** | 可以将标签添加到架构中。 数据集中的所有字段都会继承架构的标签。 |
| **字段标签** | 字段标签是从架构继承或直接应用于字段的数据治理标签。 应用于字段的数据治理标签不会继承到架构级别。 |
| **地理围栏** | 地理围栏是由GPS或RFID技术定义的虚拟地理边界，它使软件能够在移动设备进入或离开特定区域时触发响应。 |
| **身份标签** | “I”身份标签用于对可以识别或联系特定人员的数据进行分类。 |
| **基于兴趣的定位** | 如果满足以下三个条件，则会发生基于兴趣的定位（也称为个性化）：在网站上收集的数据用于推断用户的兴趣，而在其他上下文中使用，例如在另一个网站或应用程序（站外）上，该数据用于根据这些推断选择提供哪些内容或广告。 |
| **营销活动** | 在数据治理框架上下文中，营销行动是指 [!DNL Experience Platform] 数据使用者需要使用，为此，需要检查是否存在违反数据使用策略的情况 |
| **策略** | 在数据治理框架中，策略是一种规则，用于描述允许或不允许对特定数据执行哪种营销操作。 |
| **架构标签** | 在架构级别管理数据治理、同意和访问控制的标签。 这会将标签传播到使用该架构的每个数据集。 |
| **敏感标签** | “S”敏感标签用于对您和您的组织认为敏感的数据进行分类。 |
