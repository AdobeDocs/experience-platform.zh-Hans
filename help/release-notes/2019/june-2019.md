# Adobe Experience Platform 发行说明

**发行日期：2019 年 6 月 28 日**

Adobe Experience Platform的新增功能：

* [[!DNL Data Science Workspace]](#dsw)
* [[!DNL Decisioning Service]](#decisioning)
* [[!DNL Query Service]](#query)

现有功能的更新：

* [[!DNL Experience Data Model (XDM)]](#xdm)
* [[!DNL Segmentation Service]](#segmentation)

## [!DNL Data Science Workspace] {#dsw}

Adobe Experience Platform [!DNL Data Science Workspace] 是 [!DNL Experience Platform] 通过构建和运行机器学习模型，数据科学家能够跨Adobe解决方案和第三方系统无缝地从数据和内容生成洞察。 [!DNL Data Science Workspace] 与 [!DNL Platform] 并支持端到端数据科学的生命周期，包括探索和准备XDM数据，随后开发和实施模型以自动丰富 [!DNL Real-Time Customer Profile] 和机器学习分析。

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 预配和计算隔离 | 提供所需的专用计算资源，使数据科学家能够在 [!DNL Experience Platform] 以安全的方式。 |
| 首次用户体验 | 包括各种机器学习框架和语言(例如 [!DNL Python]、R、PySpark和Scala [!DNL Spark]. |
| 笔记本 | 由提供技术支持的数据科学家/数据工程师的自定义环境 [!DNL Jupyter Notebooks] 以使他们能够准备数据、提取功能，并开发包含精选库和常用机器学习框架列表的ML模型。 |
| 数据探索 | 无缝访问摄取到的XDM数据 [!DNL Platform] 集成 [!DNL Platform Data Access] SDK。 |
| 数据可视化 | 能够在中执行SQL查询 [!DNL Jupyter Notebooks] 加快数据准备和功能工程。 |
| 特征管线 | 适用于Scala/PySpark的API/SDK可部署功能工程管道，以将核心XDM数据转换为功能架构。 |
| 模型创作 | 模板和运行时，使数据科学家能够专注于模型开发，而无需实施用于访问数据和计算资源的基础架构代码。 您可以导入模型代码并将其操作，以便从中的数据派生分析 [!DNL Platform]. |
| 企业模型管理 | 支持多租户数据模型来跟踪模型版本和关联的超参数配置，以便为合作伙伴生态系统奠定基础。 |
| 模型评估 | 在 [!DNL Python]、PySpark、R和Scala。 |
| 模型部署 | 能够比较多个实验运行中的评估量度和配置，并作为服务发布最佳模型。 |
| 批量评分 | 丰富 [!DNL Real-Time Customer Profile] ，或将它们作为数据集写回 [!DNL Platform] |
| 计划 | 与集成 [!DNL Platform] 编排服务，通过API使用用户定义的计划自动化模型培训、评分和功能管道。 |

**已知问题**

* 计划和功能管道当前仅通过API提供，未来版本中将添加UI。

有关更多信息，请访问 [数据科学工作区概述](../../data-science-workspace/home.md).

## [!DNL Decisioning Service] {#decisioning}

Adobe Experience Platform [!DNL Decisioning Service] 提供了以编程方式智能地从给定个人的一组可用选项中选择“下一个最佳体验”、将其交付到任何渠道或应用程序，以及执行报告和分析的功能。

预建的丰富数据模型能够以与渠道无关的方式支持“下一个最佳选件”决策的用例。

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 业务对象存储库 | 由JSON模式模型驱动的存储库允许开发人员创建、读取、更新和删除各种业务对象。 存储库提供通用、含意见的查询API以及模式感知搜索。 |
| 存储库容器 | 在业务对象存储库中，开发人员可以将他们对项目、业务或组织单位或项目生命周期阶段（例如，在开发与集成、暂存或实时生产使用中）的关注点隔离开来。 这些隔离称为存储库容器。 |
| 角色和权限 | 使用 [!DNL Admin Console]，组织可以创建和管理用户档案，以按类型、访问操作和容器授予对资源的目标访问权限。 可以将用户添加到这些访问配置文件中，并自动从这些策略计算有效的访问权限。 |
| 预建选件对象模型 | 无需先构建数据模型， [!DNL Platform] 开发人员可以利用预建的JSON架构和关系创建选件目录、定义决策规则和约束，以及组合用于决策的选件集。 |
| 基于用户档案和非用户档案数据的决策规则 | 与 [!DNL Real-Time Customer Profile] 允许开发人员创建利用用户档案数据的决策规则。 不仅可以使用用户档案属性做出决策，而且还可以基于用户档案的体验事件历史记录以及与用户身份无关的业务实体（例如，流量条件、产品清单）做出决策。 任意 [!DNL Experience Data Model] (XDM)实体中存在架构 [!DNL Schema Registry] 可用于决策规则。 规则是第一类实体，可重复用于任何决策选项和活动。 |
| 排名和上限 | 对满足给定用户的所有资格和其他限制的决策选项进行排名，并选择最佳选项。 可以使用附加的每个用户和全局上限约束来限制可用选项的显示，从而实现资源约束和用户疲劳的个性化。 |
| [!DNL Decisioning] REST API | 的 [!DNL Decisioning Service] 可以使用简单的REST API调用，以获取给定个人的下一个最佳选件。 量度API可用于检查实时选件建议和上限值。 |
| 将决策事件流式传输到 [!DNL Data Lake] 和 [!DNL Query Service] | 的 [!DNL Decisioning Service] 自动创建数据集，以自动将所有XDM决策事件流式传输到 [!DNL Data Lake]. 然后，可以使用数据集进行分析和报告 [!DNL Query Service]. |
| 开发人员支持 | 自助服务选择加入，其中包含有关Adobe I/O的文档，其中包含有关各种主题的教程。 |

**已知问题**

* 选件数据模型不会通过 [!DNL Schema Registry] 因此，只能以有限的方式进行扩展。 模型架构具有内置结构，允许附加自定义数据。 将来，您将能够扩展基本XDM模型类，以定义自己的自定义决策域。
* 您必须使用选件管理域模型进行配置，并且必须在此产品上下文中管理用户和集成。

## [!DNL Query Service] {#query}

[!DNL Query Service] 提供使用标准SQL在Adobe Experience Platform中查询数据的功能，以支持许多不同的分析和数据管理用例。 它是一个无服务器工具，允许您连接 [!DNL Data Lake] 并将查询结果捕获为新数据集以用于报表， [!DNL Data Science Workspace]或 [!DNL Profile Service].

您可以使用 [!DNL Query Service] 构建数据分析生态系统，创建消费者在其各种交互渠道中的全景图。 这些渠道可能包括：

* 销售点系统
* Web
* 移动设备
* CRM系统

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 查询编辑器 | 使用基于Web的工具编写、验证、测试和执行查询。 它包括一个控制台，用于提供有关查询执行的详细信息以及预览查询结果的功能。 |
| 数据集创建 | 在上创建数据集 [!DNL Experience Platform] 通过标准SQL语法。 |
| Adobe定义的函数 | 利用快捷方式功能执行常见任务，如识别会话或设置归因。 |
| BI工具连接 | 使用常见BI工具中的PostgreSQL(Postgres)驱动程序连接到 [!DNL Query Service] 创建报表和可视化图表。 支持的工具包括： [!DNL Tableau], [!DNL Power BI]和 [!DNL Looker]. 在“凭据”选项卡上查找身份验证信息。 |
| 数据库管理工具连接 | 连接 [!DNL Aqua Data Studio] 或 [!DNL DB Visualizer] to [!DNL Query Service] 用于数据探索和数据集创建功能。 [!DNL Query Service] 还支持从R Studio连接。 在“凭据”选项卡上查找身份验证信息。 |
| 命令行查询工具 | 连接PSQL以能够从命令行运行查询。 |
| 查询日志 | 保留由执行的查询的历史记录 [!DNL Query Service] 和允许您查找以前的SQL，以便编辑、执行或在结果之外创建数据集。 |
| 查询计划API | 计划通过此API定期执行的查询。 |

**已知问题**

* [!DNL Query Editor] 显示查询结果的100行示例。 要保留完整的结果集，请使用查询日志中的数据集创建功能。
* 近期版本将添加对视图的支持和将计划应用到查询的UI。

有关 [!DNL Query Service]，请参阅 [产品文档](../../query-service/home.md).

## [!DNL Experience Data Model] (XDM) {#xdm}

标准化和互操作性是其背后的关键概念 [!DNL Experience Platform]. [!DNL Experience Data Model] (XDM)在Adobe的驱动下，努力标准化客户体验数据并定义客户体验管理架构。

XDM是一项公开记录的规范，旨在提高数字体验的强大功能。 它为任何与Adobe Experience Platform上的服务通信的应用程序提供了通用结构和定义。 通过遵循XDM标准，所有客户体验数据都可以整合到通用呈现形式中，以更快、更集成的方式提供洞察。 您可以从客户操作中获得有价值的分析，通过区段定义客户受众，以及将客户属性用于个性化目的。

XDM是允许 [!DNL Experience Cloud]由Adobe Experience Platform提供支持，在适当的时间通过适当的渠道向适当的人员传递适当的信息。

方法 [!DNL Experience Platform] 是建的， [!DNL XDM System] 操作 [!DNL Experience Data Model] 供使用的架构 [!DNL Experience Platform] 组件。

**新增功能**

| 功能 | 描述 |
| ---------- | ------------ |
| JSON架构约束 | 以下数据类型现在在用户界面中具有其他用于定义约束的选项： `string`  — 最小/最大长度，模式，默认值，格式(定义 [JSON架构草稿–6](https://tools.ietf.org/html/draft-wright-json-schema-01))和 `double`  — 最小值/最大值，默认值。 |
| 自定义 `$id` | 您现在可以提供自己的 `$id` 值。 |
| 架构注册表性能改进 | 优化了并集模式生成，并增强了模式缓存，从而大大缩短了API响应时间。 |

**错误修复**

* 将identityMap字段从上下文/配置文件移出并移入其自己的架构字段组，以使定义标识更加直观。
* 已使用context/identitymap根据上下文/配置文件修补所有现有架构。
* 修复了未提供版本时的错误消息。
* 修复了以下错误： [!DNL Schema Registry] 正在为用户档案合并架构调用提供随机响应。
* 修复了合并架构无法在 [!DNL Schema Registry].
* 修复了偶尔无法使用有效命名空间创建身份描述符的错误。
* 修复了对象使用 `properties` 而不是 `allOf`.

**已知问题**

* 无法扩展 [!DNL Platform] — 提供的字段组（通过添加字段）。
* 从架构组合中删除字段组时，不会删除描述符。
* 无法创建没有标签的枚举字段。

要了解有关使用XDM的更多信息，请使用 [!DNL Schema Registry] API和 [!DNL Schema Editor]，请阅读 [XDM系统文档](../../xdm/home.md).

## [!DNL Segmentation Service] {#segmentation}

[!DNL Segmentation Service] 从用户档案存储中定义特定的用户档案子集，用于描述区分用户档案存储中可销售人群的标准。 区段可以基于记录数据（如人口统计信息）或代表客户与您品牌接触点的时间系列事件。

例如，在以跑鞋为重点的电子邮件促销活动中，您可以使用受众区段，该区段包含过去30天内搜索跑鞋但未完成购买的所有用户。 另一个示例是使用区段来定位网站内容，以便该内容仅显示给属于您奖励计划特定层级的访客。

**新增功能**

| 功能 | 描述 |
| -----------| ---------- |
| 相对时间规则 | 您现在可以选择滚动时间窗口，如14天前、3到5小时前等。 |
| XDM字段概要 | 对于左边栏中的“属性”，现在提供了概要，以便您查看基础数据。 |
| 左边栏搜索 | 改进了左边栏区段部分的搜索功能。 |
| eVar友好名称 | 改进了对友好名称的支持，使您能够更轻松地查看在来自Adobe Analytics的自定义事件和维度中捕获的信息。 |
| 合并策略支持 | 现在，您可以使用简单的下拉菜单选择要应用于其区段定义的合并策略。 |

**错误修复**

* 修复了导致左边栏中属性和事件构建块加载缓慢的间歇性问题。
* 修复了导致估算器返回“NaN”响应的错误。
* 修复了某些字段打开错误的规则构建画布的错误。

**已知问题**

* None.

有关更多信息，请参阅 [Segmentation Service概述](../../segmentation/home.md).
