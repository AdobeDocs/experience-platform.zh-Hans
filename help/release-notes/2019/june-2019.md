Adobe Experience Platform的新增功能：

* [数据科学工作区](#dsw)

## 数据科学工作区 {#dsw}

Adobe Experience Platform Data Science Workspace是Experience Platform中的一项完全托管服务，它使数据科学家能够通过构建和操作机器学习模型，跨Adobe解决方案和第三方系统无缝地从数据和内容生成洞察。 数据科学工作区与平台紧密集成，支持端对端数据科学的生命周期，包括探索和准备XDM数据，然后开发并实施模型，以利用机器学习洞察自动丰富实时客户用户档案。

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 配置和计算隔离 | 提供专用计算资源，使数据科学家能够以安全方式在Experience Platform内执行不受信任的代码。 |
| 首次用户体验 | 包括针对各种机器学习框架和语言（如Python、R、PySpark和Scala Spark）的现成范例。 |
| 笔记本 | 为Jupyter Notebooks支持的数据科学家／数据工程师定制环境，使他们能够准备数据、提取功能并使用库和流行的机器学习框架的精选列表开发ML模型。 |
| 数据探索 | 无缝访问引入与Platform Data Access SDK集成的平台中的XDM数据。 |
| 数据可视化 | 能够在Jupyter笔记本中执行SQL查询以加速数据准备和功能设计。 |
| 特征管线 | 用于Scala/PySpark的API/SDK部署功能工程管道，将核心XDM数据转换为功能模式。 |
| 模型创作 | 使数据科学家能够专注于模型开发而无需实施用于访问数据和计算资源的基础架构代码的模板和运行时。 您可以导入模型代码并将其操作，以便从平台中的数据派生洞察。 |
| 企业模型管理 | 支持多租户数据模型以跟踪模型版本和相关的超参数配置，为合作伙伴生态系统提供基础。 |
| 模型评估 | 在Python、PySpark、R和Scala中评估和优化回归和分类模型。 |
| 模型部署 | 能够比较多个实验运行中的评估指标和配置，并将最佳模型作为服务发布。 |
| 批量评分 | 利用机器学习洞察丰富实时客户用户档案，或将它们作为数据集写回平台 |
| 计划 | 与Platform Orcheration Service集成，通过API通过用户定义的计划自动化模型培训、评分和功能管道。 |

**已知问题**

* 计划和功能管道目前仅通过API可用，并且UI将添加到将来的发行版中。

有关详细信息，请访 [问Data Science Workspace概述](../../data-science-workspace/home.md)。

## 决策服务 {#decisioning}

Adobe Experience Platform Decisioning Service能够以编程方式智能地为特定个人从一组可用选项中选择“下一个最佳体验”，将其交付到任何渠道或应用程序，并执行报告和分析。

预建的丰富数据模型以渠道不可知的方式支持“下一个最佳优惠”决策的用例。

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 业务对象存储库 | 由JSON模式模型驱动的存储库允许开发人员创建、读取、更新和删除各种业务对象。 存储库提供通用、具有表现力的查询API以及模式感知搜索。 |
| 存储库容器 | 在业务对象存储库中，开发人员可以将他们的疑虑隔离在项目、业务或组织单元周围，或围绕项目的生命周期阶段（例如，在开发和集成、分阶段或用于实时生产）。 这些隔离称为存储库容器。 |
| 角色和权限 | 使用Admin Console，组织可以创建和管理用户档案，以按类型、访问操作和容器授予对资源的目标访问权限。 可以将用户添加到这些访问用户档案，并根据这些策略自动计算有效的访问权限。 |
| 预建优惠对象模型 | 无需首先构建数据模型，平台开发人员就可以利用预建的JSON模式和关系创建优惠目录、定义决策规则和约束以及为决策组合优惠集合。 |
| 基于用户档案和非用户档案数据的决策规则 | 与实时客户用户档案紧密集成，使开发人员能创建利用用户档案数据的决策规则。 不仅可以使用用户档案属性，而且还可以基于用户档案的体验事件历史以及基于与用户标识无关的业务实体（例如，流量条件、产品库存）做出决策。 模式注册表中存在模式的任何体验数据模型(XDM)实体都可用于决策规则。 规则是第一类实体，可以重复用于任何决策选项和活动。 |
| 排名和上限 | 对满足给定用户的所有资格和其他约束的决策选项进行排名，并选择最佳选项。 可使用附加的每用户和全局限制约束限制来限制可用选项的暴露，从而实现具有资源约束和用户疲劳的个性化。 |
| 决策REST API | 决策服务可以使用简单的REST API调用，以获得给定个人的“下一个最佳优惠”。 度量API可用于检查实时优惠建议和上限值。 |
| 流决策事件到数据湖和查询服务 | 决策服务自动创建数据集，以自动将所有XDM决策事件流化到数据湖中。 然后，使用查询服务可以对数据集进行分析和报告。 |
| 开发人员支持 | 自助服务选择加入有关Adobe I/O的文档，其中包含有关各种主题的教程。 |

**已知问题**

* 优惠数据模型不通过模式登记处公开，因此只能以有限的方式扩展。 模型模式具有内置结构，允许附加自定义数据。 将来，您将能够扩展基本XDM模型类以定义您自己的自定义决策域。
* 您必须配置优惠管理域模型和用户，并且必须在此产品上下文中管理集成。

要了解更多信息，请访 [问决策服务概述](../../decisioning-service/home.md)。

## 查询服务 {#query}

查询服务提供使用标准SQL在Adobe Experience Platform中查询数据的能力，以支持许多不同的分析和数据管理用例。 它是一个无服务器工具，它允许您加入Data Lake中的任何数据集，并将查询结果捕获为新数据集以用于报告、数据科学工作区或引入用户档案服务。

您可以使用查询服务来构建数据分析生态系统，从而创建消费者在各种交互渠道中的形象。 这些渠道可能包括：

* 销售点系统
* Web
* 移动设备
* CRM系统

**主要功能**

| 功能 | 描述 |
| -----------| ---------- |
| 查询编辑器 | 使用基于Web的工具编写、验证、测试和执行查询。 它包含一个控制台，用于提供有关执行查询的详细信息以及预览查询结果的能力。 |
| 数据集创建 | 通过标准SQL语法在Experience Platform上创建数据集。 |
| Adobe定义的函数 | 利用快捷键功能实现常见任务，如标识会话或设置归因。 |
| BI工具连接 | 使用常见BI工具中的PostgreSQL(Postgres)驱动程序连接到查询服务以创建报告和可视化。 支持的工具包括：Tableau、Power BI和Looker。 在“凭据”选项卡上查找身份验证信息。 |
| 数据库管理工具连接性 | 将Aqua Data Studio或DB可视器连接到查询服务，以便进行数据探索和创建数据集功能。 查询服务还支持R Studio的连接。 在“凭据”选项卡上查找身份验证信息。 |
| 命令行查询工具 | 连接PSQL以能够从命令行运行查询。 |
| 查询日志 | 保留由查询服务执行的查询的历史记录，使您能够查找以前的SQL，以便编辑、执行或在结果中创建数据集。 |
| 查询计划API | 计划查询通过此API重复执行。 |

**已知问题**

* 查询编辑器显示了100行的查询结果示例。 要保留完整的结果集，请使用查询日志中的数据集创建功能。
* 近期发行版将增加对视图的支持和将计划应用于查询的UI。

有关查询服务的详细信息，请参阅产 [品文档](../../query-service/home.md)。

## 体验数据模型(XDM) {#xdm}

标准化和互操作性是Experience Platform的主要概念。 Adobe推动的体验数据模型(XDM)旨在标准化客户体验数据并定义客户体验管理模式。

XDM是一个公开的规范，旨在提高数字体验的强大功能。 它为与Adobe Experience Platform上的服务通信的任何应用程序提供了通用结构和定义。 通过遵守XDM标准，所有客户体验数据都可以整合到以更快、更集成的方式提供洞察的共同表现形式中。 您可以从客户行动中获得宝贵的洞察，通过细分定义客户受众，并将客户属性用于个性化目的。

XDM是一种机制，它允许Experience Cloud以Adobe Experience Platform为后盾，在正确的时间在正确的渠道上向正确的人员提供正确的信息。

XDM系统构建Experience Platform的方法，可操作Experience Data Model模式，供Experience Platform组件使用。

**新增功能**

| 功能 | 描述 |
| ---------- | ------------ |
| JSON模式限制 | 以下数据类型现在在用户界面中具有定义约束的其他选项： `string` -最小／最大长度、模式、默认值、格式(如 [JSON模式draft-6中定义](https://tools.ietf.org/html/draft-wright-json-schema-01))和 `double` -最小／最大默认值。 |
| 自定义 `$id` | 现在，在POST请求中创 `$id` 建资源时，您可以提供自己的价值。 |
| 模式注册表性能改进 | 优化了合并模式生成，并增强了模式缓存，从而大大缩短了API响应时间。 |

**错误修复**

* 将identityMap字段从上下文/用户档案移出并移入其自己的混合中，以使定义身份更直观。
* 基于上下文/用户档案和上下文/identitymap修补所有现有模式。
* 修复了未提供版本时的错误消息。
* 修复了模式注册表为用户档案合并模式调用提供随机响应的错误。
* 修复了合并模式未在模式注册表中显示正确字段的错误。
* 修复了标识描述符有时无法使用有效命名空间创建的错误。
* 修复了对象使用而不是使用的取消 `properties` 引用问题 `allOf`。

**已知问题**

* 无法通过添加字段来扩展平台提供的混音。
* 当从模式合成中删除混音时，描述符不被删除。
* 无法创建不带标签的枚举字段。

要了解有关使用模式注册表API和模式编辑器使用XDM的更多信息，请阅读 [XDM系统文档](../../xdm/home.md)。

## 分段服务 {#segmentation}

细分服务定义用户档案商店中特定的用户档案子集，描述区分用户档案商店中可销售人群的标准。 细分可以基于记录数据（如人口统计信息）或表示客户与您品牌接触点的时间序列事件。

例如，在专注于跑鞋的电子邮件活动中，您可以使用过去30天内搜索跑鞋但未完成购买的所有用户的受众段。 另一个示例是使用区段来目标网站内容，以便仅显示给属于您奖励项目的特定层的访客。

**新增功能**

| 功能 | 描述 |
| -----------| ---------- |
| 相对时间规则 | 您现在可以选择滚动时间窗口，如14天前、3到5小时前等。 |
| XDM字段摘要 | 对于左边栏上的“属性”，现在提供摘要，为基础数据提供视图。 |
| 左边栏搜索 | 改进了左边栏的段部分的搜索功能。 |
| eVar友好名称 | 改进了对友好名称的支持，使您能更轻松地从Adobe Analytics的自定义事件和维度中捕获哪些信息。 |
| 合并策略支持 | 您现在可以使用简单的下拉菜单选择要应用于其区段定义的合并策略。 |

**错误修复**

* 修复了导致左边栏中属性和事件构建块加载缓慢的间歇性问题。
* 修复了导致估计器返回“NaN”响应的错误。
* 修复了某些字段打开错误规则构建画布的错误。

**已知问题**

* 无.

有关详细信息，请参阅 [Segmentation Service概述](../../segmentation/home.md)。
